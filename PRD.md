## 1. 产品愿景与定位

### 一句话定义
**WorkMemory 是一个住在你桌面上的工作记忆助手——它默默记住你做过的一切，在你需要的时候帮你回忆、整理、输出。**

### 产品定位
- **不是**任务管理工具（不要用户主动填写待办）
- **不是**笔记工具（不要用户主动记录）
- **是**自动化的工作记忆外挂：用户只管工作，WorkMemory 自动在后台帮你记
- **核心价值**：把"我今天做了什么"从脑力消耗变成零成本获取

### 竞品差异化

| 维度 | 其他工具 | WorkMemory |
|------|---------|-----------|
| 记录方式 | 用户主动输入 | 全自动后台截图+OCR |
| 存在感 | 图标藏在任务栏 | 桌面伙伴，有温度，随时可见 |
| 输出 | 原始笔记 | AI 整理的工作报告 |
| 隐私 | 云端同步 | 100% 本地，数据不离机 |
| 情感 | 冰冷的工具 | 有个性的伙伴 |

## 2. 核心设计哲学（不可妥协）

### 哲学一：感知优先
> 用户感受到的好用 > 功能数量 > 技术先进性

- 每个功能上线前，问：**用户第一眼看到这个会觉得"哇"还是"嗯"？**
- 宁可少一个功能，也不要有一个体验糟糕的功能
- 动画、过渡、细节打磨不是装饰，是产品的一部分

### 哲学二：零负担原则
> 使用 WorkMemory 本身不应该消耗用户的认知资源

- 后台运行，不打扰用户工作主流程
- 所有主动操作都应该在 30 秒内完成
- 绝不强迫用户填写表单、打标签、分类整理

### 哲学三：有温度的存在
> WorkMemory 不是一个冷冰冰的软件，是一个有个性的工作伙伴

- 桌面伙伴是产品的"脸"，它的每个动作都应该让用户感到被关心
- 提醒要有人情味，不是系统通知，是朋友在说话
- 错误提示也要温暖，不要技术报错语言

### 哲学四：好看是刚需
> 颜值不是加分项，是基本要求

- 界面精致度对标 Linear、Raycast、Arc 浏览器
- 每个组件都应该经得起放大看
- 动画流畅度 60fps，过渡自然，不生硬

## 3. 目标用户

### 主要用户：知识工作者
**典型画像**：
- 每天同时处理多个项目、多个窗口
- 需要每周写工作报告但总是不记得做了什么
- 经常被人问"上周那个事情进展怎么样了"，脑子里一片空白
- 在意隐私，不愿意数据上云

**核心痛点**：
1. 写日报/周报要花 20-30 分钟回忆
2. 找之前做过的东西要翻很久
3. 和别人复盘某个项目时想不起细节
4. 不知道自己时间花在哪里了

### 次要用户：自由职业者 / 创作者
- 需要向客户证明工作量
- 想了解自己的专注规律
- 需要积累工作成果作为作品集素材

## 4. 体验原则

### 视觉原则
1. **主题**：清爽、现代、略带温暖感，不冰冷科技风，不花哨
2. **色彩**：以白/浅灰为底，主题色用于强调，不超过 2 个主色
3. **字体**：系统字体优先（苹方/微软雅黑），保持可读性
4. **圆角**：适度圆角（8-12px），不要直角生硬，不要过度圆
5. **阴影**：柔和阴影，体现层次感，不要扁平无深度，不要夸张

### 交互原则
1. **响应即时**：点击后 100ms 内有视觉反馈
2. **过渡自然**：页面切换、状态变化都有平滑动画（200-300ms）
3. **容错友好**：误操作可撤回，删除操作需确认
4. **加载优雅**：数据加载时有骨架屏，不要白屏闪烁
5. **空状态有内容**：列表为空时展示引导内容，不要空白一片

### 信息密度原则
1. **今日页面**：信息丰富但不拥挤，一眼能看出今天做了什么
2. **细节按需展开**：默认展示摘要，点击展开详情
3. **不强制分类**：AI 自动归类，用户可以修改但不强制

## 5. 产品模块详细规格

---

### 5.1 桌面伙伴（Mascot）— 核心灵魂

> **定位**：WorkMemory 的"脸"和入口，用户与产品的情感连接点。
> 它应该让用户感到：这个软件是活的，它在关心我。

#### 5.1.1 基本形态

**窗口特性**：
- 独立透明窗口，始终置顶（always-on-top）
- 默认尺寸：80×80px（可在设置中调整 60-120px）
- 位置：屏幕右下角，距边缘 20px（可拖拽，拖拽后吸附到最近边缘）
- 无标题栏、无边框、无背景色
- 非伙伴本体区域点击穿透

**拖拽吸附规则**：
- 拖拽松手后，以 300ms 动画吸附到屏幕四个角之一（就近原则）
- 记住最后位置，重启后恢复

#### 5.1.2 外形系统（5 种形象）

每种形象有独立的角色人格，影响气泡措辞风格和动画表现：

| 形象 ID | 昵称 | 性格 | 气泡语气 |
|---------|------|------|---------|
| `note` | 备忘录小鸟 | 认真、负责、爱整理 | 专业温暖 |
| `film` | 胶片小熊 | 文艺、感性、善回忆 | 诗意柔和 |
| `copilot` | 宇航员猫 | 高效、专业、技术范 | 简洁直接 |
| `cursor` | 光标精灵 | 活泼、灵动、好奇 | 俏皮活泼 |
| `paper` | 折纸狐狸 | 智慧、温和、博学 | 睿智从容 |

**每种形象需要的设计资源**（UI 设计完成后提供给 AI）：
- 静止帧（PNG，透明背景，80×80px）
- 6-8 帧循环动画（推荐 Lottie JSON 或逐帧 PNG）
- 专属主题色（用于气泡边框和高亮）

**设计暂用方案**：在设计稿未完成之前，每种形象用对应 emoji + 圆形背景色代替，功能完整，等设计出图后替换资源文件即可。

#### 5.1.3 状态系统

**功能状态**（影响图标视觉）：

| 状态 | 触发条件 | 视觉表现 |
|------|---------|---------|
| `recording` | 正在记录 | 左下角绿色小圆点 + 慢速呼吸光晕 |
| `paused` | 用户手动暂停 | 灰度滤镜 + 暂停图标角标 |
| `privacy` | 检测到无痕窗口 | 遮眼动作（捂眼/蒙眼帘）+ 紫色调 |
| `ocr_scanning` | OCR 识别中 | 眼睛扫描动画（左右移动光线）|
| `report_ready` | 今日报告可生成 | 金色高亮光晕 + 角标数字"1" |

**情绪状态**（叠加在功能状态之上，影响动画速度和气泡语气）：

| 情绪 | 触发条件 |
|------|---------|
| `happy` | 生成了完整报告 / 知识库新增条目 |
| `focused` | 检测到用户连续专注 20 分钟以上 |
| `concerned` | 用户连续工作超过 2 小时未休息 |
| `curious` | 检测到从未见过的新应用或新项目 |
| `sleepy` | 系统时间 23:00 以后 / 系统长时间闲置 |
| `proud` | 周目标完成度 > 80% |
| `neutral` | 默认状态 |

#### 5.1.4 微动画规格

所有动画必须流畅（60fps），不卡顿：

- **呼吸动画**（录制中）：scale 1.0 → 1.04 → 1.0，周期 3 秒，ease-in-out
- **漂浮动画**（空闲/中性）：translateY 0 → -5px → 0，周期 4 秒，ease-in-out
- **扫描动画**（OCR 中）：眼睛部位左右扫描，周期 1.5 秒
- **弹跳动画**（report_ready）：scale 1.0 → 1.15 → 0.95 → 1.05 → 1.0，单次触发
- **入场动画**（首次显示）：从屏幕底部飞入，落地时轻微弹跳
- **点击反馈**：按下 scale 0.92，松开 scale 1.0，100ms
- **拖拽状态**：轻微旋转 ±5°，松手后弹回吸附边缘

#### 5.1.5 气泡通知系统

**气泡视觉规格**：
- 磨砂玻璃效果背景（backdrop-filter: blur 16px）
- 白色半透明底 rgba(255,255,255,0.88)
- 柔和阴影（0 8px 32px rgba(0,0,0,0.12)）
- 圆角 16px，最大宽度 280px，内边距 16px
- 出现：scale 0.8 + opacity 0 → scale 1.0 + opacity 1，弹性曲线 250ms
- 消失：淡出下移，200ms
- 默认展示 6 秒后自动消失（有操作按钮的气泡不自动消失）

**气泡三种模式**：

模式 1 · 纯文字提醒（自动消失）：

    ┌────────────────────────────────┐
    │  今天已专注工作 3.5 小时 💪    │
    └────────────────────────────────┘

模式 2 · 带操作按钮（需用户交互，不自动消失）：

    ┌────────────────────────────────┐
    │  📋 今日报告已准备好了！        │
    │  要查看一下吗？                │
    │                                │
    │  [查看报告]      [稍后提醒]    │
    └────────────────────────────────┘

模式 3 · 今日摘要卡片（悬停触发）：

    ┌────────────────────────────────┐
    │  今天 · 6月22日                │
    │  ──────────────────────────    │
    │  🔵 10:30  写代码 · 重构模块   │
    │  🟢 11:45  开会 · 产品评审     │
    │  🔵 14:00  文档整理            │
    │                                │
    │  专注 3.5h · 事件 6 个        │
    │  [打开完整记忆]                │
    └────────────────────────────────┘

#### 5.1.6 交互行为

**鼠标悬停（mouseEnter）**：
- 伙伴轻微放大（scale 1.0 → 1.08），100ms
- 250ms 后展示今日摘要卡片（模式 3 气泡）

**鼠标离开（mouseLeave）**：伙伴恢复原始大小，摘要卡片淡出消失

**左键单击**：
- 如果有未读报告 → 打开主窗口并跳转到报告页
- 如果有未完成待办 > 0 → 展示今日待办气泡
- 否则 → 打开主窗口并跳转到 Today 页面

**右键单击（弹出上下文菜单）**：

    📅 打开今日记忆
    📝 快速记一笔...
    ⏸  暂停记录  /  ▶ 继续记录（切换）
    📋 生成今日报告
    ──────────────
    ⚙️  设置
    👁  隐藏 10 分钟

**右键双击**：弹出"快速捕获"输入框（Ghost Capture），回车保存，ESC 取消

**拖拽**：
- 按住左键可拖动伙伴，拖拽中轻微旋转 ±5°
- 松手后以 300ms 动画吸附到屏幕四个角之一（就近原则）
- 位置跨重启持久化

#### 5.1.7 主动提醒规则

**时间驱动（后台定时）**：
- 每天 17:30：「今天工作即将结束，生成今日报告？」（附操作按钮）
- 每周五 17:00：「本周的工作报告可以整理了 📋」
- 每天 09:00：「新的一天开始！昨天完成了 N 件事 ✨」

**行为驱动（实时检测）**：
- 连续专注同一应用 >= 25 分钟 → 「专注了 25 分钟，可以休息一下 ☕」
- 5 分钟内窗口切换 >= 10 次 → 「今天切换比较频繁，要专注一件事吗？」
- 连续工作 >= 2 小时无休息 → 「已连续工作 2 小时了，起来活动一下吧 🚶」
- 夜间 22:00 仍在工作 → 「都 22 点了，注意休息哦 🌙」

**事件驱动**：
- 检测到从未见过的新项目名 → 「发现了新项目：{项目名}，要了解一下吗？」
- 知识库待审核积累 >= 3 条 → 「有 N 条知识等待你确认」
- AI 生成新技能卡 → 「你解锁了一个新技能 🏆」

**频率限制**：
- 同类提醒最短间隔 30 分钟；每天提醒总次数上限 8 次
- 用户设置的免打扰时段内不发送任何提醒

#### 5.1.8 首次启动引导

1. 伙伴从屏幕右下角底部飞入（入场动画），落地后轻微弹跳
2. 气泡依次出现（每条 3 秒，点击可跳过）：
   - 「嗨！我是你的工作记忆助手 👋」
   - 「我会在后台悄悄记录你的工作，你只管专心做事就好」
   - 「每天结束前，我来帮你整理今天做了什么」
   - 「点我随时看今日记忆，右键查看更多功能」
   - 「现在来选一个你喜欢的形象？」（展示 5 种形象供选择）
3. 用户选择形象后，伙伴变换外形并弹跳一下表示高兴
4. 引导结束，开始正常后台记录

#### 5.2.5 右侧详情面板

- 回车即保存为手动 Episode（source: 'manual'）

#### 5.2.7 待办汇总区

    □ 修复登录页 bug

- 来源：AI 从 OCR 文字中自动提取
- 支持手动添加；完成的待办划线但当日内保留

### 5.3 记忆采集层（后台）

> 用户感受不到这一层，但它决定了记忆质量。核心原则：**多记，记准，不打扰**。

#### 5.3.1 截图采集
- 后台定间隔（默认 30 秒）检测窗口变化
- **感知哈希去重**：相邻帧相似度 > 90% 时跳过 OCR（预计减少 60-70% 无效推理）
- 截图分辨率：长边不超过 1280px（平衡 OCR 精度和性能）
- 检测到无痕模式时立即停止截图，切换为占位符记录

#### 5.3.2 OCR 识别
- **OCR 必须在独立 Worker Thread 中运行，绝不阻塞主进程事件循环**
- 同内容不重复识别（哈希去重后只有变化内容进队列）
- OCR 失败时记录原因，不影响整体流程
- 支持模型切换：tiny（快速）/ small（精准）

#### 5.3.3 剪贴板监听
- 监听用户复制操作（clipboard 内容变化事件）
- 经隐私规则过滤（密码管理器、银行 app 等应用跳过）
- 复制内容与当前活跃 Segment 关联存入 metadata
- 超过 5000 字符的内容只取前 500 字符 + 标注「内容过长」

#### 5.3.4 隐私保护
- 无痕浏览器窗口：自动检测，立即停止截图，桌面伙伴进入 privacy 状态（遮眼动画）
- 用户自定义隐私规则：按应用名 / 窗口标题关键词 / URL 关键词匹配，完全跳过不记录
- 所有截图本地存储，绝不上传任何数据到云端
- 用户可在 Settings 随时查看「今日哪些内容被隐私保护了」

### 5.4 Episode 理解层（后台）

> 将原始截图 + OCR 文字，变成有意义的"工作事件"。

#### 5.4.1 Episode 自动生成

每段连续工作自动归并为一个 Episode，包含：
- **标题**：AI 总结（5-15 字，简洁描述做了什么）
- **一句话摘要**（30-60 字）
- **时间段**（开始 - 结束）
- **涉及实体**：项目、人物、文档
- **待办事项**（从 OCR 文字中提取，存入 todos 列表）
- **阻塞项**（存入 blockers 列表）
- **类型分类**：work / meeting / research / coding / planning / reading / communication

#### 5.4.2 跨天任务连续性
- 每天生成 Episode 后，自动检测与过去 7 天 Episodes 的关联性
- 关联判断：项目名相同、实体重叠 >= 2 个、或 AI 语义相似度 > 0.8
- 建立 relatedEpisodeIds 字段，Today 页面展示「昨天也在做这件事」提示

#### 5.4.3 待办提取规则

OCR 文字出现以下模式时自动提取为待办：
- 「TODO / todo / 待办 / 待处理 / 需要 / 下一步」后面的内容
- Markdown 格式 `- [ ] 内容`
- 编辑器中的 TODO 注释
- 会议记录中出现的 Action Item

### 5.5 搜索（Search）

> 体验要像用 Google，不像用数据库。

#### 5.5.1 搜索输入
- 顶部全宽搜索框，始终可见
- 输入即时出结果（300ms 防抖）
- 支持自然语言，不要求精确关键词
- 快捷过滤语法：`#编码` `@张三` `>上周` `project:XX项目`

#### 5.5.2 搜索结果
两种模式（顶部 Tab 切换）：
- **时间顺序**：适合「当时发生了什么」
- **相关度排序**：适合「找某个主题的所有记录」

每条结果展示：标题 + 时间 + 关键词高亮 + 匹配维度说明

#### 5.5.3 实体时间线视图

#### 5.5.4 自然语言时间搜索

- 「上周五下午」→ 自动转换为时间范围过滤
- 「上个月做了什么」→ 展示上月 Episodes 摘要
- 「最近一次和张三开会」→ 找到最近 meeting 类型 + @张三 的 Episode

## 7. 技术架构规格

> 以下是给 AI 的技术选型规格。项目从零构建，AI 自行决定最优实现细节。

### 7.1 技术栈选型

| 层级 | 技术选型 | 说明 |
|------|---------|------|
| 桌面框架 | **Electron 31+** | Windows 优先，支持系统托盘/全局快捷键/透明窗口 |
| 构建工具 | **Vite 5** | 开发热更新，生产构建 |
| 前端框架 | **React 18** | 函数组件 + Hooks |
| 类型系统 | **TypeScript**（strict 模式）| 全覆盖，不用 any |
| 状态管理 | **Zustand** | 轻量全局状态，按模块拆分 |
| UI 组件库 | **Radix UI** | 无样式基础组件（Dialog / Popover / Select…）|
| 动画库 | **Framer Motion** | 页面过渡、组件动画 |
| 本地数据库 | **better-sqlite3** | WAL 模式，FTS5 全文检索 |
| IPC 校验 | **Zod** | 所有 IPC 入参必须有 Zod schema 校验 |
| OCR | **PaddleOCR**（koffi FFI 调用）| OCR 必须在 Worker Thread 中运行 |
| 测试 | **Vitest** | 核心逻辑单元测试 |
| 打包 | **electron-builder** | NSIS Windows 安装包 |

### 7.2 核心架构约束

**IPC 设计**：
- 所有通道以 `模块名:操作名` 命名，如 `capture:start` `episode:list`
- 入参统一 Zod schema 校验
- 返回值统一信封：`{ ok: true, data: T }` 或 `{ ok: false, error: string }`
- Preload 层明确标注所有入参和返回类型，禁止使用 `unknown`

**数据库**：
- SQLite WAL 模式，应用退出前执行 `PRAGMA wal_checkpoint(TRUNCATE)`
- 使用版本号管理 Schema 迁移（PRAGMA user_version），逐版本 up 迁移
- 新增字段通过迁移追加，不覆盖初始 Schema

**安全**：
- `sandbox: true`（渲染进程沙箱化）
- `contextIsolation: true` + `nodeIntegration: false`
- API Key 使用系统 keychain（safeStorage）加密存储，不明文写入数据库
- 生产环境 CSP 不包含 localhost 地址

**性能**：
- OCR 推理在独立 Worker Thread，绝不阻塞主进程和 UI
- 感知哈希截图去重，相似度 > 90% 跳过 OCR
- 启动到主窗口可交互时间 < 3 秒

**日志**：
- 统一日志模块，支持写入文件（app.getPath('userData')/logs/）
- 生产环境不向控制台输出 debug 信息
- 错误日志包含：时间戳 / 模块 / 错误类型 / 堆栈

### 7.3 数据库目标 Schema

> AI 根据以下表结构设计进行实现，字段类型和约束请根据实际需求补充。

**核心表**：
- `segments`：原始截图片段（id, timestamp, ocr_text, window_title, app_name, image_path, ocr_blocks, ocr_confidence, capture_source, active_window_bounds…）
- `episodes`：工作事件（id, date, start_time, end_time, title, summary, type, project, entities, topics, todos, blockers, segment_ids, source…）
- `clean_episodes`：AI 精炼后的事件（含 memory_kind, confidence, wiki_eligible, related_episode_ids…）
- `wiki_pages`：知识卡片（id, title, type, content, backlinks, last_cited_at, status…）
- `reports`：生成的报告（id, date, report_type, content, template_id, created_at…）
- `privacy_rules`：隐私保护规则（id, rule_type, pattern, enabled…）
- `distill_runs`：AI 精炼任务记录
- `weekly_patterns`：周级工作模式分析结果
- `skill_cards`：AI 提炼的技能卡
- `user_goals`：用户设定的周目标（新增）

**FTS5 虚拟表**（全文搜索）：
- `fts_segments`：索引 segments.ocr_text + window_title
- `fts_episodes`：索引 episodes.title + summary
- `fts_wiki`：索引 wiki_pages.content

### 7.4 中期演进方向

考虑 v2.0 迁移到 **Tauri v2** + Rust 后端：
- 内存从 ~300MB 降至 ~40MB，安装包从 ~100MB 降至 ~8MB
- 前端 React 代码完全不变
- 探索 Windows 内置 OCR API（零额外内存，调用系统服务）
